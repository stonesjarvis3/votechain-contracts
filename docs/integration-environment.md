# Integration Environment

This document explains how to provision and use the reproducible integration environment for contract and API tests.

## Overview

The integration environment spins up:

| Service | Image | Purpose |
|---|---|---|
| `stellar-node` | `stellar/quickstart:latest` | Local Stellar network with Soroban RPC |
| `redis` | `redis:7-alpine` | Backend cache layer |

Contracts are built from source and deployed to the local network before tests run. The environment is fully disposable — every run starts from a clean state.

## Quick Start

### Prerequisites

- Docker and Docker Compose (or Docker Desktop)
- Rust toolchain with `wasm32-unknown-unknown` target
- Stellar CLI (`cargo install --locked stellar-cli --features opt`)

### 1. Copy the environment file

```bash
cp .env.integration.example .env.integration
```

The defaults work out of the box for local development. You only need to edit `.env.integration` if you want to point at a different network.

### 2. Start services and deploy contracts

```bash
./scripts/provision-integration-env.sh
```

This script:
1. Starts `stellar-node` and `redis` via Docker Compose
2. Waits until the Stellar RPC is ready
3. Builds WASM binaries
4. Deploys governance and token contracts to the local network
5. Writes the resulting contract IDs back into `.env.integration`

### 3. Run integration tests

```bash
# Contract integration tests
source .env.integration
cargo test --test '*' -- --nocapture

# Backend integration tests (uses Redis + contract IDs from .env.integration)
cd backend && npx vitest run
```

### 4. Tear down

```bash
./scripts/provision-integration-env.sh --down
```

This stops all services and removes Docker volumes so the next run starts clean.

### 5. Reset (tear down + re-provision)

```bash
./scripts/provision-integration-env.sh --reset
```

## Using Docker Compose Directly

If you prefer to control services individually:

```bash
# Start only the infrastructure (stellar node + redis)
docker compose -f docker-compose.integration.yml up -d stellar-node redis

# Run the full stack including deployer and test runner
docker compose -f docker-compose.integration.yml up --exit-code-from test-runner

# Tear down
docker compose -f docker-compose.integration.yml down --volumes --remove-orphans
```

## CI Usage

GitHub Actions runs integration tests automatically on every push to `main` and `develop` and on pull requests targeting those branches. See [`.github/workflows/integration-tests.yml`](../.github/workflows/integration-tests.yml).

The CI workflow:
1. Starts `stellar-node` and `redis` as service containers
2. Installs Rust + Stellar CLI, restoring caches between runs
3. Generates a throwaway deployer key
4. Builds contracts and deploys to the local network
5. Runs `cargo test --test '*'` and `vitest run` in the backend

No secrets are required for integration CI — the local network generates accounts automatically.

## Environment Variables

| Variable | Default | Description |
|---|---|---|
| `NETWORK` | `local` | Network identifier passed to deploy scripts |
| `STELLAR_RPC_URL` | `http://localhost:8000/soroban/rpc` | Soroban RPC endpoint |
| `STELLAR_NETWORK_PASSPHRASE` | `Standalone Network ; February 2017` | Must match the quickstart node's passphrase |
| `STELLAR_SECRET_KEY` | *(generated)* | Deployer account secret; auto-generated by provision script |
| `REDIS_URL` | `redis://localhost:6379` | Backend Redis connection |
| `VOTECHAIN_GOVERNANCE_CONTRACT_ID` | *(populated after deploy)* | Set by provision script after deployment |
| `VOTECHAIN_TOKEN_CONTRACT_ID` | *(populated after deploy)* | Set by provision script after deployment |

## Troubleshooting

**`stellar-node` never becomes healthy**

The quickstart image takes 10–20 s to initialise. The provision script retries for 80 s (40 × 2 s). If it still times out, check Docker memory limits — the node needs at least 1 GB.

```bash
docker compose -f docker-compose.integration.yml logs stellar-node
```

**`cargo test` fails with "contract not found"**

The deploy step may have failed silently. Check the deploy output:

```bash
cat /tmp/votechain-deploy.log
```

Then re-run the provision script:

```bash
./scripts/provision-integration-env.sh --reset
```

**Redis connection refused**

Ensure Redis is running:

```bash
docker compose -f docker-compose.integration.yml ps redis
```

**Port conflicts**

If ports 8000 or 6379 are already in use locally, stop the conflicting service or change the host port mappings in `docker-compose.integration.yml`.
